## Problem Statement

GS-Tracker Phase 1 已经能自动抓取 Goldman Sachs 13F 报告、生成中文 AI 分析报告并部署为 Web 服务。但报告生成后，用户必须主动打开网站才能看到更新；系统不会主动告诉用户“新季度报告已出”。

对于希望把高盛动向作为投资决策参考的用户来说，漏看一个季度报告意味着错过重要的调仓信号。因此需要在报告生成后，自动把摘要推送到用户常用的消息渠道。

## Demand Evidence

- 上一版设计文档已将“新季度报告生成后自动推送通知（邮件/飞书）”列为 MVP 必备项。
- 用户明确表示 GS-Tracker 的目标是“做成产品”，需要面向真实用户持续可用。
- 用户当前状态是“手动查看”报告，这是通知功能要解决的核心痛点。

## Status Quo

目标用户目前获取新报告的方式是手动检查：
- 打开浏览器访问 GS-Tracker Web 服务
- 在报告列表里查看有没有新季度
- 没有主动提醒机制

代价是：
- 容易漏掉新季度披露（13F 有 45 天滞后，披露当天往往被市场快速消化）
- 需要记住每周/每季度去检查
- 无法第一时间把报告分享到投资讨论群

## Target User & Narrowest Wedge

**目标用户**：有一定投资经验、关注聪明钱动向、使用飞书/钉钉等协作工具的中文个人投资者。

**最小可用版本（MVP）**：
1. 新季度 13F 报告成功生成后，自动向飞书 Webhook 发送一条中文消息。
2. 消息包含：季度、总持仓市值、新增/清仓/大幅增减持数量、报告链接。
3. 通知发送失败不阻断主流程，最多请求 3 次（首次 + 2 次重试）后记录日志。
4. 同一季度只通知一次，避免重复生成或修正文件导致刷屏。
5. 末尾附带“不构成投资建议”免责声明。

## Constraints

- **技术栈**：Python 3.11+, httpx（已用于 SEC 请求）
- **渠道**：第一版仅支持飞书 Webhook；邮件/钉钉/Telegram 后续扩展
- **语言**：所有用户可见文案必须为中文
- **安全**：Webhook URL 通过 `.env` 注入，禁止硬编码或提交到 git
- **稳定性**：通知失败不得影响报告生成和 Web 服务
- **幂等性**：同一季度只发送一次通知，使用 SQLite 表 `sent_notifications` 记录
- **部署**：通知逻辑随 scheduler 容器一起运行
- **现有代码**：仓库已存在 `src/notifier.py`（含 `Notifier` 类与多通道占位），本设计在现有类上实现飞书，不另起炉灶

## Premises

1. 第一版通知只支持飞书 Webhook，其他渠道后续扩展。
2. 触发时机：`src/main.py` 成功生成一份**新季度** 13F 报告后自动推送。
3. “新季度”以数据库 `sent_notifications` 表无该季度记录为准；文件生成前通知函数不会收到调用。
4. 飞书 Webhook 使用普通群机器人模式（无签名验证）；企业版自定义签名需求后续扩展。
5. 通知内容用中文，包含季度摘要、关键变化和报告链接。
6. 飞书 Webhook URL 使用现有 `.env` 变量名 `FEISHU_WEBHOOK`，代码里不硬编码。
7. 通知发送失败时，对 5xx / 429 / 网络超时最多请求 3 次（首次 + 2 次重试），退避间隔 1s、2s；4xx 客户端错误（除 429）不重试；最终失败记录日志，不阻断主流程。
8. 第一版忽略 13F 修正文件（13F-HR/A）的再次通知；修正文件覆盖数据后，用户可通过 Web 服务查看最新报告。
9. 通知末尾加上“本内容仅供学习参考，不构成投资建议”。
10. 报告链接使用新配置项 `PUBLIC_BASE_URL` 拼接；若未配置，通知中省略链接并记录警告。

## Approaches Considered

### Approach A: 在现有 Notifier 类中实现飞书
- 仓库已有 `src/notifier.py` 和 `Notifier` 类，以及 `FEISHU_WEBHOOK` 配置。
- 实现 `_send_feishu(notification: Notification)`，将 `Notification` 对象转成飞书文本消息 JSON 并发送。
- `main.py` 生成报告后构造 `Notification` 并调用 `Notifier.send()`。
- 使用 SQLite 表 `sent_notifications` 记录已通知季度，避免重复发送。

**工作量**：小（约 1 个方法 + 1 个测试文件 + 少量主流程改动）
**风险**：低
**优点**：与现有代码结构一致，不引入新抽象，最快上线
**缺点**：后续新增渠道仍需修改 `Notifier` 类，但当前成本可接受

### Approach B: 抽象接口 + 飞书实现
- 保留 `Notification` dataclass。
- 定义 `NotifierProtocol`，`FeishuNotifier`、`EmailNotifier` 等分别实现。
- `Notifier` 变成分发器，组合多个 channel。

**工作量**：中
**风险**：低
**优点**：扩展新渠道时无需改动调用方
**缺点**：当前只有一个真实渠道，抽象层收益不明显；与现有占位代码结构冲突

### Approach C: 创意方向 — 飞书交互卡片
- 使用飞书机器人卡片消息（带颜色、按钮、字段布局）。
- 比纯文本更醒目，但需要维护 JSON 卡片模板。

**工作量**：中
**风险**：中
**优点**：视觉冲击强，适合群里分享
**缺点**：格式要求严格，调试成本高，不如纯文本通用

## Recommended Approach

**选择 Approach A：在现有 `Notifier` 类中实现飞书。**

理由：
- 仓库已经为通知模块留出位置，`FEISHU_WEBHOOK` 变量和 `Notification` dataclass 都已存在。
- 直接实现 `_send_feishu` 是最小改动路径，符合 YAGNI。
- Phase 1 刚完成，先让通知跑通、验证用户价值，再考虑多渠道抽象。
- 与上一版设计文档中“优先实现一个通知渠道（飞书）”保持一致。

## Open Questions

1. 飞书 Webhook URL 是否已申请好？
2. 未来扩展到邮件/钉钉时，是同一消息多渠道发送，还是按用户偏好单渠道发送？
3. 是否需要“只有发生重大持仓变化才通知”的阈值开关？

## Success Criteria

- [ ] 新季度报告生成后，飞书群收到一条中文通知消息
- [ ] 通知包含报告季度、关键变化摘要和可点击的报告链接（PUBLIC_BASE_URL 配置时）
- [ ] 通知末尾包含“不构成投资建议”免责声明
- [ ] `FEISHU_WEBHOOK` 未配置时，系统静默跳过通知，不报错
- [ ] 飞书接口返回 5xx/429/超时时，最多请求 3 次；4xx 客户端错误不重试；最终失败记录日志但不阻断主流程
- [ ] 同一季度重复生成时，不会重复发送通知
- [ ] `tests/test_notifier.py` 覆盖成功发送、失败重试、4xx 不重试、未配置跳过、重复去重五种情况
- [ ] 所有测试通过：`pytest -v`

## Distribution Plan

- **访问方式**：用户通过飞书群消息接收通知。
- **CI/CD**：沿用现有 GitHub 仓库 + `/ship` 流程；无需新增发布管道。

### 实现 `_send_feishu`

```python
async def _send_feishu(self, notification: Notification) -> None:
    text = notification.title
    if notification.body:
        text += f"\n\n{notification.body}"
    if notification.link:
        text += f"\n\n查看完整报告：{notification.link}"
    text += "\n\n本内容仅供学习参考，不构成投资建议。"

    payload = {
        "msg_type": "text",
        "content": {"text": text},
    }

    response = await self.client.post(
        FEISHU_WEBHOOK,
        json=payload,
        timeout=20.0,
    )
    response.raise_for_status()
    data = response.json()
    if data.get("code") != 0:
        raise httpx.HTTPStatusError(
            f"Feishu error: {data.get('msg')}",
            request=response.request,
            response=response,
        )
```

### 重试策略

使用 `src/config.py` 中已有的 `ANTHROPIC_MAX_RETRIES` / `ANTHROPIC_BACKOFF_BASE` 模式，新增：

```python
NOTIFIER_MAX_ATTEMPTS = int(os.getenv("NOTIFIER_MAX_ATTEMPTS", "3"))
NOTIFIER_BACKOFF_BASE = float(os.getenv("NOTIFIER_BACKOFF_BASE", "1.0"))
```

在 `_send_feishu` 外层加循环（或装饰器）：

```python
for attempt in range(1, NOTIFIER_MAX_ATTEMPTS + 1):
    try:
        await self._send_feishu_once(notification)
        return
    except httpx.HTTPStatusError as exc:
        code = exc.response.status_code if exc.response else 0
        if 400 <= code < 500 and code != 429:
            raise
        if attempt == NOTIFIER_MAX_ATTEMPTS:
            raise
        await asyncio.sleep(NOTIFIER_BACKOFF_BASE * (2 ** (attempt - 1)))
    except httpx.TransportError:
        if attempt == NOTIFIER_MAX_ATTEMPTS:
            raise
        await asyncio.sleep(NOTIFIER_BACKOFF_BASE * (2 ** (attempt - 1)))
```

`run_pipeline()` 中对 `Notifier.send()` 的调用用 `try/except Exception` 捕获并记录日志，不阻断主流程。

### 构造 `Notification` 的摘要统计

在 `src/main.py` 的 `run_pipeline()` 中，生成报告后、发送通知前：

```python
import pandas as pd
from src.notifier import Notification, Notifier
from src.quarter_compare import QuarterComparator
from src.storage import get_holdings

previous_quarter = "2025-Q4"  # 实际从数据库查询上一季度
prev_records = get_holdings(cik, previous_quarter)
summary = None
if prev_records:
    prev_df = pd.DataFrame(prev_records)
    comparison = QuarterComparator().compare(df, prev_df, quarter, previous_quarter)
    summary = {
        "total_value": float(df["value"].sum()),
        "new_positions": len(comparison.new_positions),
        "sold_positions": len(comparison.sold_positions),
        "increased_positions": len(comparison.increased_positions),
        "decreased_positions": len(comparison.decreased_positions),
    }
```

`df["value"]` 已是美元单位（数据获取时已将千美元乘以 1000）。

若 `prev_records` 为空（首次运行），`summary` 为 `None`，通知函数跳过变化统计，只发送“报告已生成”和链接。

### 阈值说明

通知文案中“大幅增减持”沿用 `QuarterComparator` 默认阈值：**季度持仓市值变化 ≥ 20%**。文案可写为“大幅（变化≥20%）增持/减持”，让用户明确含义。

### 去重机制

`src/storage.py` 新增表：

```sql
CREATE TABLE IF NOT EXISTS sent_notifications (
    quarter TEXT PRIMARY KEY,
    sent_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

新增函数：

```python
def mark_notification_sent(quarter: str) -> bool:
    """Atomically mark a quarter as notified.

    Returns True if this call actually inserted the row (i.e. first time),
    False if the quarter was already present.
    """
    with get_connection() as conn:
        try:
            conn.execute(
                "INSERT INTO sent_notifications (quarter) VALUES (?)",
                (quarter,),
            )
            conn.commit()
            return True
        except sqlite3.IntegrityError:
            return False
```

发送流程：

```python
if storage.mark_notification_sent(quarter):
    notifier = Notifier()
    try:
        await notifier.send(notification)
    except Exception:
        logger.exception("Failed to send notification for %s", quarter)
    finally:
        await notifier.close()
else:
    logger.info("Notification already sent for %s; skipping", quarter)
```

`INSERT OR IGNORE` 原子性避免并发重复发送。

### 金额格式化

`src/notifier.py` 中新增 `_format_value(value: float) -> str`：

```python
def _format_value(value: float) -> str:
    if value >= 1e9:
        return f"${value/1e9:.1f}B"
    if value >= 1e6:
        return f"${value/1e6:.1f}M"
    if value >= 1e3:
        return f"${value/1e3:.1f}K"
    return f"${int(value)}"
```

### 消息格式化

def _format_summary(summary: Optional[dict]) -> str:

        f"新增持仓：{summary['new_positions']} 只\n"

## Reviewer Concerns

本次 adversarial review 完成 3 轮迭代。以下问题已解决或明确：
- 与现有 `Notifier` 类、`FEISHU_WEBHOOK` 变量名、`QuarterComparator` 20% 阈值对齐。
