## Problem Statement

GS-Tracker Phase 1 只从 SEC 13F 季报获取高盛持仓信号。但 13F 有 45 天披露延迟——高盛在 3 月底的调仓，用户要 5 月中旬才能看到。信号准确但滞后。

当前用户手动去 Bloomberg、Reuters、财联社搜索高盛相关新闻来弥补这个信息缺口。这是一个典型的"手工胶带拼凑"工作流——用户每天都在做，但效率低、易遗漏、缺乏结构化分析。

**核心命题**: 把 13F 持仓数据 + 新闻事件 + SEC 8-K 重大事件披露聚合到同一个视图，让 AI 做交叉验证，产生单一数据源无法提供的信号洞察。

## Demand Evidence

- 用户已经在手动做这件事——每天去 Bloomberg/Reuters/财联社搜高盛相关新闻
- 不满足于"新闻 AI 摘要推飞书"，明确要求信号交叉验证视图（Q4 选择了"全量信号仪表盘"）
- 用户愿意接受更大的构建范围来获得完整的信号视图（方案选择 B，跳过渐进式方案 A）

## Status Quo

用户当前工作流：
1. 打开 GS-Tracker HTML 报告看季度 13F 分析
2. 自行去 Bloomberg、Reuters、财联社手动搜索高盛和高盛持仓公司相关新闻
3. 在人脑中做交叉验证——"高盛增持了 Apple，同时 Apple 刚发了财报，这两件事之间有关系吗？"

没有工具帮他们做这层关联。整个过程靠人脑、靠记忆、靠运气。

## Target User & Scope

- **用户画像**: 关注高盛投资动向的个人投资者/研究员，有一定金融知识，但不一定有专业终端（Bloomberg Terminal）
- **范围**: 多源信号聚合引擎——同时接入新闻 + SEC 8-K（重大事件披露），嵌入现有 HTML 报告。SC 13G（5%+ 持股申报）关注的是高盛持仓公司的其他大股东变动，而非高盛自己申报，作为 Phase 2.5 延后
- **排除**: 不接入付费数据源，不做实时股价/期权异动（成本太高，Phase 3 再考虑）

## Constraints

1. **数据源只使用免费/公开的**（SEC EDGAR、RSS 新闻源）。不依赖付费 API
2. **AI 做质量过滤而非人力审核**——靠 Claude API 判断信号质量和相关性
3. **交付方式不变**——信号面板嵌入现有 HTML 报告，用户打开同一份报告就能看到
4. **代码内部英文、用户可见中文**
5. **TDD 开发**——每个模块先写失败测试，再写实现
6. **YAGNI**——不做多余的抽象层，不做"以后可能需要"的扩展点

## Premises

1. **信号交叉验证创造独特价值**: 13F 持仓 + 新闻事件 + SEC 文件在同一视图里的交叉验证，能产生单一信源无法提供的洞察。**用户确认**。

2. **免费数据源 + AI 过滤可行**: SEC EDGAR 免费提供 8-K（重大事件披露）等结构化 XML 文件。SC 13G（5%+ 持股申报）数据也可获取，但留到后续迭代。RSS 新闻源免费但需要 AI 做质量过滤和去重。**用户选择此策略**。

3. **嵌入现有报告优于新建渠道**: 多源信号面板嵌入已有的 `{quarter}.html` 报告，而非新建独立仪表盘或仅通过飞书推送。用户已经习惯打开 HTML 报告——在同一页面增加信号层，零学习成本。**用户确认**。

## Approaches Considered

### Approach A: 渐进式——先加新闻验证信号交叉价值

**Summary**: 只改造 `news_monitor.py`（已有 stub），加入 RSS 新闻抓取 + AI 摘要，在 HTML 报告中新增"新闻信号"模块。不做 SEC 文件监控，不做统一信号框架。

**Effort**: S（2-3 天）
**Risk**: 低
**Pros**:
- 最快验证"信号交叉是否产生价值"这一核心假设
- 只改一个已有 stub + reporter 模板，改动范围极小
- 失败了可以快速回滚，不污染架构
**Cons**:
- 用户已明确拒绝此方案——认为单源新闻不能满足需求
- 后续加 SEC 文件源需要重复改造 reporter 和分析层
**Reuses**: `news_monitor.py` stub、`reporter.py` Jinja2 模板机制

### Approach B: 全量信号框架——多源聚合引擎 ✅ 选中

**Summary**: 构建统一的信号聚合引擎 `SignalAggregator`，每个数据源（13F、新闻、8-K）实现统一的 `SignalSource` 接口。引擎负责并行拉取、统一格式、去重排序。Claude API 对聚合后的信号做交叉验证分析。HTML 报告重构为多面板信号仪表盘。

**Effort**: L（2-3 周）
**Risk**: 中——架构复杂度增加，多源错误处理需要仔细设计
**Pros**:
- 用户明确选择的方向——一次构建完整的信号交叉验证能力
- 统一接口让后续添加新信号源（股价异动、期权数据等）成本极低
- 信号评分引擎可以量化每条信号的强度和相关性
**Cons**:
- 测试矩阵大——每个源需要 mock，错误路径多样化
- 如果信号交叉验证的假设本身不成立，投入浪费
- Claude API 调用次数增加（每个信号源可能需要独立分析）
**Reuses**: `SEC13FFetcher` 的 httpx 异步模式、`quarter_compare.py` 的变化检测逻辑、`notifier.py` 的格式化辅助函数

### Approach C: AI 驱动合成——放弃分类信号源

**Summary**: 不做"新闻源""SEC 源"的分类架构。所有原始数据（13F + SEC XML + RSS headlines）聚合后一次性送入 Claude prompt，让 AI 做交叉分析、优先级排序、洞察提炼。代码最简洁，AI 承担分类和关联的全部工作量。

**Effort**: M（1 周）
**Risk**: 中——依赖 Claude API 的质量和 token 限制，输入数据量受上下文窗口约束
**Pros**:
- 代码极简——只需要数据拉取层 + 一个大型 Claude prompt
- 自动适应新数据源——添加新源只需在输入里多塞一段文本
**Cons**:
- Claude 上下文窗口有上限，大量数据可能截断，13F 已有 2000+ 条持仓记录
- 无法做确定性去重和排序——依赖 LLM 的一致性（不稳定）
- 用户明确要求信号交叉验证"产生真正的洞察"，纯 AI 合成缺乏可复现性
**Reuses**: GSAnalyzer 的 Claude API 调用模式

## Recommended Approach

**选择 B: 全量信号框架**。原因有三：
1. 用户三次独立决策都指向完整方案（Q4 选全量仪表盘、方案选择 B、前提 1 确认信号交叉验证是核心价值）
2. 统一接口架构让 Phase 3（股价数据、期权异动等付费源）的边际成本降到最低
3. 信号评分引擎是可测试的——不像方案 C 依赖 LLM 黑盒

### SignalSource 接口草稿

```python
from dataclasses import dataclass
from datetime import datetime
from typing import List, Protocol

@dataclass
class Signal:
    """Unified signal format across all sources."""
    title: str              # e.g. "高盛增持 Apple 5%"
    source: str             # "13F" | "news" | "8-K" | "SC 13G"
    published_at: datetime
    url: str | None         # link to source document
    summary: str            # AI-generated Chinese summary (≤200 chars)
    companies: List[str]    # tickers or issuer names mentioned
    strength: str           # "high" | "medium" | "low"
    cross_refs: List[str]   # IDs of related signals from other sources

class SignalSource(Protocol):
    """Each data source implements this interface."""
    source_name: str

    async def fetch(self, quarter: str) -> List[Signal]:
        """Pull raw data for the quarter and return normalized signals."""
        ...

    async def close(self) -> None:
        """Release HTTP client / resources."""
        ...
```

## Open Questions

1. **信号评分算法**: 信号强度如何量化？时序接近度（新闻 + 13F 调仓时间差）、相关性强度（新闻提及的具体公司 vs 13F 持仓重叠度）、来源可信度权重——这些参数需要实验确定
2. **Claude 调用策略**: 每个信号源单独调用 Claude 分析 vs 聚合后一次性分析——成本和延迟的权衡
3. **RSS 新闻源具体选型**: 哪些 RSS 源覆盖高盛和高盛持仓公司？财联社、Reuters RSS 是否还需要 API key？
4. **新闻去重**: 同一事件被多个新闻源报道，如何识别和合并？标题相似度？URL 规范？

## Success Criteria

1. HTML 报告新增"多源信号面板"模块——与 13F 分析同页展示
2. 至少 3 个信号源就绪：13F 持仓（已有）+ 新闻 + SEC 8-K
3. 每条信号有：标题、来源、时间、AI 摘要、信号强度（高/中/低）
4. AI 交叉验证分析：当新闻事件与 13F 持仓变化有时间/主体重叠时，自动标记为"交叉信号"并生成解释
5. 所有 63 个现有测试继续通过
6. 新增信号源模块的测试覆盖率 ≥ 80%

## Distribution Plan

现有 Docker Compose 部署管线覆盖。新模块跟随 `scheduler.py` 的每周执行周期自动运行。前端通过已有的 `web.py` FastAPI 服务提供，HTML 报告路径不变（`{quarter}.html`）。

## Dependencies

- **SEC EDGAR API**: 已用于 13F 数据获取（`data_fetcher.py`），扩展到 8-K/SC 13G 使用同一 API，无新依赖
- **RSS 解析**: 需要添加 `feedparser` 库（纯 Python，无系统依赖）
- **Claude API**: 已集成（`analyzer.py`），用于信号摘要和交叉验证
- **HTML 模板**: 现有 Jinja2 模板（`templates/`），需要新增信号面板组件

## The Assignment

下一周做一件事：**验证"信号交叉验证"这个核心假设是否成立**。手动收集一期数据：找到高盛最近一次 13F 调仓 + 同期的 3-5 条相关新闻 + 1-2 份 SEC 8-K 文件，扔给 Claude 做一次性的交叉分析。如果 Claude 产出的洞察比你单独看 13F 更有价值，假设成立，全速开建。如果发现信号之间其实没什么可交叉验证的，省下三周的构建时间。

## What I noticed about how you think

- 你在前提挑战里说的"信号交叉验证是核心价值"——这不是随便说的。你拒绝了渐进式方案和 AI 合成方案，三次独立决策都指向同一个方向：要的是真正的洞察质量，不是快。这种一致性很少见。
- 你选数据策略的时候立刻选了"免费源 + AI 过滤"——说明你对成本有直觉，知道什么值得花钱什么不值得。这是 builder 的财务直觉，不是每个做产品的人都有。
- "嵌入现有报告"这个选择也很有判断力——你已经有了一个用户打开的产品，不新建渠道而是增强现有入口。这是产品 sense，不是技术决策。
