## Problem Statement

有一定投资经验的个人投资者希望跟踪 Goldman Sachs 的投资动向，作为自己股票决策的参考。但现有方案要么信息太碎片（靠零散看新闻、研报、公众号），要么太专业昂贵（彭博终端、FactSet），要么只是机械的 13F 数据展示，缺乏 AI 解读和多源信号聚合。

用户需要一个**自动化的、中文的、基于多源信号的高盛动向情报系统**：把 13F 季度持仓作为基准仓位，叠加研究报告、宏观策略、交易信号等间接信息，用 AI 推断方向性意图，并以可视化的 HTML 情报板形式呈现。

## Demand Evidence

- 用户表示**已有付费意向**：目标用户愿意为高质量的高盛动向情报付费。
- 用户自己就是这个工具的强需求用户，投资决策高度依赖 Goldman Sachs 动向信息。
- 当前市场上缺乏针对中文个人投资者的、将 13F 数据与多源信号聚合解读的产品。

## Status Quo

目标用户目前主要靠**碎片信息判断**：
- 零散阅读高盛研报、宏观观点、市场新闻
- 手动访问 WhaleWisdom、13F.info 等网站查看持仓变化
- 靠自己的经验和感觉把信息拼凑成投资决策

现状方案的代价：
- **时间成本高**：每季度手动追踪、筛选、解读信息耗时数小时
- **信息噪声大**：缺乏统一框架判断哪些信号真正重要
- **决策主观**：没有系统化的多源交叉验证，容易误判

## Target User & Narrowest Wedge

**目标用户**：有一定投资经验、关注聪明钱动向、愿意为高质量情报付费的中文个人投资者。

**最小可用版本（MVP）**：
1. 自动抓取 Goldman Sachs 最新 13F-HR 报告
2. AI 生成中文持仓分析报告
3. 生成可视化 HTML 报告
4. 部署为可通过浏览器访问的 Web 服务
5. 新季度报告生成后自动推送通知（邮件/飞书）

**快速叠加阶段**（同一开发周期内）：
- 高盛研究报告采集与观点提取
- 宏观策略追踪
- 交易信号监控（大宗交易、期权异动、ETF 流向）
- 多源信号评分引擎
- 综合中文情报板

## Constraints

- **数据限制**：SEC 13F 有 45 天滞后，且不含空仓；精确实时持仓无法公开获取
- **成本限制**：初期优先使用免费/低成本数据源，付费数据源作为可选扩展
- **技术栈**：Python 3.11, FastAPI, SQLite, Docker, 京东云 ECS 4C16G
- **语言**：所有用户可见输出必须为中文
- **部署**：Docker Compose + Nginx，长期 Web 服务形态

## Security & Compliance

- **API 密钥**：所有密钥（ANTHROPIC_API_KEY、SMTP、飞书 Webhook）通过 `.env` 注入容器，不提交到 git，不打包进镜像
- **容器安全**：Dockerfile 使用非 root 用户运行应用，使用最小化 `python:3.11-slim` 镜像
- **服务器安全**：ufw 仅开放 22/80/443，启用 fail2ban 和 unattended-upgrades
- **合规声明**：所有报告和界面必须包含"本工具仅供学习参考，不构成投资建议"的免责声明

## Error Handling & Fallbacks

- **SEC EDGAR 失败**：指数退避重试 3 次；若仍失败，记录日志并在下次调度周期重试
- **Claude API 失败**：重试 3 次，失败时生成"AI 分析暂不可用"占位文本，不阻断报告生成
- **XML 解析失败**：捕获异常，使用备用解析逻辑（如正则兜底），并告警人工检查
- **13F 修正文件（13F-HR/A）**：检测 accession number 和 filing date 变化，支持覆盖更新历史数据
- **数据源缺失**：研报/宏观/交易信号源不可用时，降级为仅展示 13F 分析，并在报告中标注信号缺失
- **容器崩溃**：Docker `restart: unless-stopped` + systemd 开机自启

## 13F Change Rules

- **新增持仓**：上一季度未出现，本季度出现
- **清仓持仓**：上一季度出现，本季度未出现
- **大幅增持**：本季度 value 较上季度增加 ≥ 25% 且绝对增加 ≥ 10 万美元
- **大幅减持**：本季度 value 较上季度减少 ≥ 25% 且绝对减少 ≥ 10 万美元
- **小幅调整**：变化在 ±10% 到 ±25% 之间
- **持仓不变**：变化在 ±10% 以内

新季度检测：每周一 09:00 CST 运行一次轮询，检查 Goldman Sachs CIK `0000886982` 的最新 13F-HR filing；使用 `(cik, accession_number)` 作为去重键。若发现修正文件（13F-HR/A），用新数据覆盖该季度历史持仓并重新生成报告。

## CUSIP & Ticker Mapping

- **主要来源**：SEC EDGAR 公司提交的 13F 信息表中包含 `nameOfIssuer` 和 `cusip`；通过 `cusip` 映射到 ticker 使用 OpenFIGI 免费 API 或维护的映射表
- **备选来源**：Yahoo Finance 搜索、`nameOfIssuer` 模糊匹配
- **无法映射**：在报告中显示公司名称，不生成交易信号；记录到 `unmapped_cusips` 表供人工补充
- **公司行为**：季度对比前，通过外部事件数据（如 Yahoo Finance actions）调整拆股、并股；合并/分拆导致的 CUSIP 变更作为已知限制标注

## Duplicate Holdings

同一 CUSIP 可能因不同 `investment_discretion`（SOLE/SHARED/OTHER）出现多条记录。处理规则：
- 按 CUSIP 聚合：`shares` 和 `value` 求和
- `investment_discretion` 保留为逗号分隔的字符串
- 投票权字段分别保留

## Signal Model（多源信号评分）

**输入信号**（权重硬编码，后续校准）：
- 13F 持仓变化（权重 0.35）：新增/大幅增持/大幅减持/清仓的仓位变化
- 研究报告观点（权重 0.25）：高盛研究部对个股/行业的评级、目标价、观点摘要
- 宏观策略一致性（权重 0.15）：高盛宏观主题（如看好科技、防御性板块）与 13F 行业配置的匹配程度
- 交易信号（权重 0.25）：持仓相关股票的大宗交易、期权异常成交量、ETF 资金流向

**宏观策略一致性计算**：
- 将宏观观点映射到 GICS 行业/板块
- 对比 13F 持仓的行业分布
- 若 13F 超配的行业与宏观看好行业一致 → bullish；反之 → bearish
- 具体规则由配置表维护，AI 负责解释不一致时的可能原因

**交易信号定义**：
- **大宗交易**：`notional_value` ≥ 100 万美元，或单日累计 ≥ 500 万美元
- **期权异动**：`volume_ratio`（期权成交量 / 过去 20 个交易日均值）≥ 2.0；统计所有行权价和到期日的看涨/看跌总成交量
- **ETF 流向**：`notional_value`（单日净流入/流出）≥ 1 亿美元

`TradingSignal` 字段根据 `signal_type` 区分：
- `block_trade` / `etf_flow`：使用 `notional_value`（美元）
- `options_flow`：使用 `volume_ratio` 和 `premium_value`（美元）

**输出格式**：
- 综合方向：`bullish` / `neutral` / `bearish`
- 置信度：0.0 - 1.0
- 贡献信号溯源：列出每个信号的方向、强度和来源

**第一阶段实现**：
- 信号权重硬编码，后续根据用户反馈手动校准（校准触发条件：累计 10 份以上用户反馈后，由维护者调整权重并对比报告质量）
- 13F 信号覆盖所有持仓股票；研报/交易信号覆盖 13F 持仓相关股票和板块
- 宏观信号覆盖 13F 持仓涉及的行业板块
- AI 负责解释评分结果；评分本身由规则+加权计算生成
- 当某个信号源缺失时，该信号权重按比例重新分配给其他可用信号：例如若研报缺失，其 0.25 权重按 0.35:0.15:0.25 的比例分配给 13F/宏观/交易信号
- 缺失信号在报告中标注"XX 信号暂不可用"

## Testing Strategy

- **单元测试**：每个 parser/fetcher/analyzer 有独立测试，使用 mock 数据
- **集成测试**：用 SEC EDGAR 真实数据跑端到端抓取（注意频率控制）
- ** golden set**：保留 1-2 份已知正确的 13F XML，作为解析回归测试
- **AI 输出评估**：人工抽查 3-5 份报告，评估中文质量、数据准确性、投资建议合理性
- **部署验证**：上线后检查 `/api/health`、报告页可访问、通知可送达

## Backup & Monitoring

- **数据备份**：宿主机每日凌晨 3 点运行 `scripts/backup.sh`，压缩 `/opt/gs-tracker/data/` 和 `/opt/gs-tracker/output/reports/` 到 `/opt/backups/gs-tracker/`，保留 30 天
- **备份恢复**：恢复时解压备份包到 `/opt/gs-tracker/` 并重启容器；恢复步骤写入 `scripts/restore.sh`
- **日志监控**：容器日志输出到 stdout/stderr，通过 `docker compose logs` 查看；关键错误发送飞书告警
- **健康检查**：Web 服务提供 `/api/health` 端点；容器异常退出时 Docker 自动重启

## Data Sources（第一阶段）

- **P0 - SEC EDGAR 13F**：免费官方数据源，Goldman Sachs CIK `0000886982`；请求头必须设置 `User-Agent`
- **P1 - 高盛公开研究**：
  - `https://www.goldmansachs.com/intelligence`（公开观点聚合）
  - `https://www.goldmansachs.com/insights`（市场洞察）
  - 第三方转载：Seeking Alpha、Bloomberg、Reuters 中关于 Goldman Sachs research 的文章
  - 实现方式：RSS/网页解析，提取标题、摘要、情感、相关行业
- **P1 - 宏观策略**：
  - `https://www.goldmansachs.com/intelligence` 中宏观主题文章
  - 实现方式：定期抓取宏观主题页面，提取观点摘要和行业映射
- **P2 - 交易信号**（第一阶段优先实现一个子源，其他为可插拔 stub）：
  - **期权异动**（优先）：CBOE 公开日终数据或免费财经 API（如 Yahoo Finance options 数据），计算成交量是否超过 20 日均值 2 倍
  - **大宗交易**（stub）：FINRA ADF 需要付费/授权，第一阶段用 mock 接口占位
  - **ETF 流向**（stub）：etf.com 或免费 API，第一阶段用 mock 接口占位
- **P3 - 新闻舆情**（可选）：Bloomberg/Reuters RSS、Seeking Alpha 等公开 RSS，作为辅助信息不进入评分模型

> **说明**：付费数据源在第一版作为可选扩展；核心功能在免费数据源下仍可运行。每个信号源独立模块，缺失时自动降级。

## Premises

1. 目标用户是愿意为高盛动向情报付费或持续使用的中文个人投资者。
2. 产品定位是"多源信号聚合的方向性意图推断"，不是"复制实时持仓"。
3. AI 聚合多源信号后输出的中文情报板，要比用户自己零散判断更清晰、更可执行。
4. 实施路径：先做 13F 核心闭环，再在同一周期内快速叠加研报、宏观、交易信号。
5. 京东云 ECS 4C16G + Docker Compose + Nginx 是合适的生产部署方案。

## Approaches Considered

### Approach A: 最小可行 — 仅 13F 核心
- 内容：SEC 13F 抓取 → AI 持仓分析 → 中文 HTML 报告 → Web 服务 → 通知
- 周期：约 2-3 周
- 风险：低
- 优点：最快上线，验证核心数据流和部署流程
- 缺点：差异化弱，只是"自动化的 13F 分析器"

### Approach B: 先做核心再快速叠加多源信号
- 内容：13F 核心闭环 → 研报采集 → 宏观策略 → 交易信号监控 → 多源信号评分 → 综合情报板
- 周期：约 6-8 周
- 风险：中
- 优点：真正体现"多源信号聚合"的差异化价值，与目标一致
- 缺点：数据源更多，错误处理、成本、稳定性更复杂

### Approach C: 平台化全功能
- 内容：多机构追踪、实时告警、付费订阅、用户系统、API 开放、社区
- 周期：3 个月以上
- 风险：高
- 优点：完整产品，可商业化
- 缺点：远离当前可验证的核心价值，容易做大量没人用的功能

## Recommended Approach

**选择 Approach B：先做核心再快速叠加多源信号。**

理由：
- 用户明确表示研报、宏观、交易信号要"真正实现"，而不仅是后续规划
- "先做核心再快速叠加"既保证了早期交付（13F 闭环可独立运行），又能在同一周期内形成差异化竞争力
- Approach A 差异化不足，Approach C 周期和风险过高

**风险缓解**：
- 13F 核心闭环必须端到端可用后再叠加其他信号，确保每个阶段都有可交付物
- 研报/宏观/交易信号采用**可插拔设计**，每个信号源独立模块，缺失时不影响整体运行
- 付费数据源使用 mock/占位实现先行验证接口，确认可用后再接入真实 API
- 优先实现一个通知渠道（飞书），其他渠道后续扩展
